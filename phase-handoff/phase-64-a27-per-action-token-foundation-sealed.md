## 主要設計判断

1. **purpose は WHERE 述語で必須強制** (advisor blocker 指摘): optional/app側後判定は禁止。SELECT+UPDATE 両方に畳み込み (defense-in-depth)。これが per-action token の全セキュリティ価値
2. **expectedPurpose は必須 positional 引数** (optional opts では必須化できず fail-fast にならない)
3. **mismatch は not_found に落とす** (別 reason を作らない = 無効 token と区別不能が正しい)
4. **A.27 は foundation のみ** (modify/cancel 発行・route は A.28)
5. **Codex sandbox-blocked → Claude 引き取り**: 2 回目委任 (service+tests) が read-only sandbox + approval-never で apply_patch 拒否 (1 回目 e2e 委任は成功、per-invocation 差)。設計は advisor + Codex read で二重検証済のため Claude 自実装 + sandbox-blocked override

## A.28 着手時の最初の判断 (要設計)

modify/cancel token の発行タイミングと route:
- **発行**: 予約作成時に view/modify/cancel token を同時発行するか、cancel は on-demand 発行か (advisor flag: 全 token を作成時に同一メールで送ると 1 通漏洩 = cancel 能力。**on-demand 発行 or 破壊的操作前の re-confirm を A.28 で決定**)
- **route**: `/r/[token]/modify`, `/r/[token]/cancel` (GET-safe: フォーム表示は consume せず loadTokenStatus(expectedPurpose) のみ、mutating POST で consume)
- **consume + mutate 同一 tx** (advisor): cancel の reservation 更新失敗で used_at がロールバックされること
- **cancel は audit_logs 必須** (advisor): action='update' + after_json.kind='customer_cancel'/'customer_modify'
- **#1 gate 発火**: 新 session 機構なし・raw-migration なしだが、destructive な未認証操作 + 新 cross-tenant boundary → adversarial enumerate 推奨
- **GET-safe 静的検査拡張**: `customer-r-token-get-safe.test.ts` を新 route file に拡張
