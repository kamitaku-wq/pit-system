### 設計判断 (なぜ TS-only / migration 不要か)

- **store_confirmed_at の書込経路**: `cancelTransportOrder` と同じく **service_role `db` (@/lib/db/client)** を admin action 経由で使う (ADR-0010 admin 経路、確立済みパターン)。service_role は RLS / column GRANT を bypass するため、vendor/authenticated の column GRANT 外である store_confirmed_at を書ける。→ **新 SECURITY DEFINER RPC は不要**。
- **status_history / change_logs を書かない**: store_confirmed_at は status 変更ではない (status は 'accepted' のまま)。`transport_order_change_logs.change_type` CHECK は `('vendor_changed','datetime_changed','cancelled','recreated','rejected_reassigned')` のみで `store_confirmed` を含まない。監査は **audit_logs trigger** が UPDATE を自動記録し、通知は本 **outbox** が担う (change_log の requires_notification 経路を使わない = 二重通知リスクなし)。
- **冪等 / 楽観排他**: UPDATE は `version=expectedVersion AND store_confirmed_at IS NULL AND status_id=accepted` をガード。二重確定不可。`idempotency_key=to:{id}:store_confirmed:v{newVersion}` (spec §15.6, C.0 で追記済)。
- **C.1 auto-confirm trigger との独立性**: confirm は status_id を触らないため trigger (BEFORE UPDATE OF status_id) は発火しない。manual 案件は accept 時に trigger 非発火で store_confirmed_at が NULL のまま → 本 action の対象。auto 案件は accept 時に確定済 (store_confirmed_at 非NULL) ゆえ本 action の `store_confirmed_at IS NULL` ガードで弾かれる (= 本 action は実質 manual 専用)。

## C.3 への引き継ぎ (★ hard checkpoint 再掲)

C.1 で記録した **vendor status 書込機構 checkpoint** が C.3 で判断ポイントになる:
- vendor は column GRANT(status_id) + vendor_portal_update policy で RPC を介さず status_id を直接 UPDATE できる既存バイパスがある。
- **C.3 `completeAction` (status→completed) の書込機構を決める際、status_id を vendor GRANT から外し RPC-only にできるか判断する** (できれば accept/complete バイパスを根本封鎖)。
- C.3 完了報告も store_confirmed_at 同様に「vendor が書ける列か」を確認 (picked_up_at/delivered_at/returned_at は vendor GRANT 内 = vendor が書ける。status_id も GRANT 内)。completeAction は vendor 自身が実行するため service_role ではなく vendor session の可能性 — 要設計判断。
