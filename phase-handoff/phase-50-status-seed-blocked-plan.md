## 発見した問題 (advisor 判定 = wake-up trigger 該当)

### 1. scope 前提条件の欠落

handoff 順 #5「production status seed 経路 `createCompanyWithDefaults`」の前提:
- spec/data-model.md §18.1: per-company status seeding (各 company で transport_statuses / transitions を seed)
- 21_seed_master migration は Phase 17 で transport status 行を削除済 (multi-tenant 化のため)
- test 環境は `tests/_helpers/seed-transport-statuses.ts` で test 内 seed
- production 環境では status が seed されないため `createTransportOrderWithNotification` / `cancelTransportOrder` 等が **本番で `StatusSeedMissingError` / `CancelStatusSeedMissingError` を throw**

→ 当初 scope: `createCompanyWithDefaults` service 関数を追加 (company 作成時に status を自動 seed)

### 2. 着手前確認で判明した重大事実

production 経路で **`companies` テーブルに company を INSERT する関数自体が存在しない**:

- `grep "createCompany|companies.insert|company.*create"` → src 配下に hit 0 件
- `tests/_helpers/seed-transport-statuses.ts` line 3 のコメント:
  > 16-E で createCompanyWithDefaults service 関数が完成したらそちらに統合し、本 helper は削除する。
- これは Phase 16-E で skip された前提

つまり Phase 50 で `createCompanyWithDefaults` を追加しても **呼び出し元 = production code path がない** ため、関数が orphan になる。

### 3. Phase 47 NO-GO 反省パターンの再現

Phase 47 で「副作用 1 件 (cancel action)」と handoff された scope が実際は副作用 5 系統で Codex adversarial review NO-GO 判定された。Phase 50 も同じパターン:
- handoff: 「production status seed (createCompanyWithDefaults 追加で MVP blocker 1 解消)」
- 実態: company 作成経路全体が未整備、status seed は **氷山の一角**

「軽微」と決め打って関数追加すると、(a) 呼び出し元なしで dead code、または (b) 呼び出し元 (admin sign-up UI / onboarding flow) も含めて大規模実装、の二択になる。
